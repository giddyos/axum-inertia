`cargo-inertia` is optional project tooling. It creates Vite frontends or
complete single-adapter projects, runs Vite beside `cargo run`, coordinates
frontend-first embedded release builds, and validates typed-page contracts.

## Install the command

```bash
cargo install cargo-inertia --version 1.0.0-alpha.1 --locked
```

## Initialize a frontend

```bash
cargo inertia init
cargo inertia init --path my-app --framework react --server-framework axum --package-manager pnpm --install --yes
cargo inertia init --path my-app --framework svelte --server-framework actix-web --package-manager bun --yes
cargo inertia init --path my-app --framework vue --server-framework rocket --package-manager npm --yes
cargo inertia init --framework react --package-manager pnpm --yes
cargo inertia init --framework svelte --package-manager bun --ssr none --yes
cargo inertia init --framework vue --package-manager npm --ssr external --ssr-endpoint http://127.0.0.1:13714 --yes
```

The final command keeps compatibility mode and creates only
`<path>/frontend`. Selecting a server framework creates a new complete project
at `--path`; it refuses to overwrite that destination. Full projects include one
Rust adapter, framework-specific handler and installation code, matching
`inertia.toml`, debug Vite configuration, and a release-only embedded provider.

The generated project:

- Pins the maintained framework versions but does not create a lockfile. Install
  dependencies and commit the resulting lockfile for reproducible builds.
- Supports CSR, managed Node SSR, and external SSR. Managed SSR needs Node 22.12+; Bun only runs package scripts and never replaces Node as the managed SSR runtime.
- Never overwrites an existing destination. Auto manager selection uses lockfiles and package metadata; conflicting lockfiles require an explicit choice.
- In frontend-only mode, creates no Rust manifest, router, handler, or root document.
- In full-project mode, never adds a direct `inertia-core` dependency or unused server adapters.

Use it as a starting point, then compare the maintained
[server framework guides](/docs/frameworks/axum) for production and SSR setup.

## Run the development processes

```bash
cargo inertia dev --frontend frontend --port 5173
```

This runs the selected npm, pnpm, or Bun `dev` script, waits for Vite, then runs Cargo with `VITE_DEV_SERVER_URL`. Cargo arguments after `--` are forwarded unchanged:

```bash
cargo inertia dev -- --bin server
```
and plain `cargo run` in the current directory. Only the Cargo child receives
`VITE_DEV_SERVER_URL=http://127.0.0.1:<port>`. When either child exits, the command
terminates its peer.

Package-manager detection supports npm, pnpm, and Bun. Cargo arguments after
`--` are forwarded to `cargo run`.

## Build a self-contained release

```bash title="Terminal"
cargo inertia build --release
cargo inertia build -- --features ssr
```

The command reads root `inertia.toml`, executes the configured frontend build,
verifies the manifest, and then invokes Cargo. The printed executable is taken
from Cargo's artifact evidence rather than assembled from assumptions. See
[Self-contained binary](/docs/guides/self-contained-binary) for configuration,
cross-compilation, binary size, and deployment checks.

## Check component declarations

```bash
cargo inertia check --path . --frontend frontend
```

The checker expects `src/main.ts`, `vite.config.ts`, and `src/Pages`. It recognizes
`.svelte`, `.tsx`, `.jsx`, and `.vue` pages and scans Rust source for literal
`#[inertia(component = "...")]` attributes.

It detects missing frontend files, duplicate Rust declarations, duplicate frontend
extensions, and unsafe component paths. It does **not** discover `page!`,
`DynamicPage::new`, or computed names; it does not reject frontend-only pages. A Vite
manifest is checked only when one already exists.

See the exact [CLI reference](/docs/reference/cli) and
[Deployment](/docs/guides/deployment) for gates that the checker does not replace.

## Synchronize Rust contracts

`cargo inertia sync` compiles typed page and props roots into TypeScript. Use `--check --deny-warnings` in CI and commit the output. The complete setup, workspace selection, layouts, integer policies, and framework imports are in [Generated TypeScript contracts](/docs/guides/generated-types).
